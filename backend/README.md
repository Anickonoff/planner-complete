# Planner Telegram Bot

Telegram-планировщик задач для группового чата с напоминаниями, дедупликацией и управлением событиями через команды.

Проект рассчитан на **один общий чат** (семейный / командный), где несколько пользователей могут добавлять и управлять событиями.

---

## Возможности

### 📌 Работа с событиями

- создание событий из текста;
- валидация (нельзя создавать события в прошлом);
- хранение в JSON (UTC);
- резервное копирование перед каждой записью.

### ⏰ Напоминания

- ежедневные напоминания о событиях на завтра;
- еженедельный список событий;
- дедупликация (одно напоминание — один раз);
- устойчивость к перезапускам.

### 📋 Команды Telegram

- `/help` — справка
- `/status` — количество событий и ближайшее
- `/today` — события на сегодня
- `/tomorrow` — события на завтра
- `/week` — события на следующую неделю
- `/delete <id>` — запрос удаления
- `/confirm <id>` — подтверждение удаления
- `/cancel` — отмена удаления

### 🌍 Часовые пояса

- хранение дат в UTC;
- отображение и сравнение в заданном TZ (например `Asia/Tomsk`);
- корректная локаль (`ru-RU`).

### 🛠 Надёжность

- long polling Telegram (без webhook);
- персистентный offset;
- дедупликация напоминаний;
- graceful shutdown;
- файловое логирование.

---

## Формат ввода событий

```
ДД.ММ [ЧЧ:ММ] Название [Описание]
```

Примеры:

```
12.01 Работа
1.2 Работа
12.01 8:30 Работа
15.01 18:30 Встреча с командой
```

Поддерживаются форматы времени:

- `8:30`
- `08:30`

Дата может быть указана как с ведущими нулями, так и без них:

- `1.2`
- `01.02`
- `7.1`
- `07.01`

## Архитектура

```
planner/
├── src/
│   ├── api/telegram/        # Telegram polling, команды
│   ├── services/            # Бизнес-логика (events)
│   ├── repositories/        # Работа с JSON
│   ├── jobs/                # cron-напоминания
│   ├── utils/               # date, logger, dedup, pending-delete
│   ├── config/              # env-конфигурация
│   └── server.js            # lifecycle сервиса
├── data/
│   ├── events.json          # хранилище
│   └── backups/             # резервные копии
├── logs/
│   └── planner.log
└── package.json
```

### Ключевые принципы

- Telegram = адаптер, не бизнес-логика
- EventsService = единый источник правил
- JSON = single source of truth
- meta = offset, дедупликация, pending-delete

## Конфигурация

Создайте файл `.env`:

```env
PORT=3000
DATA_FILE_PATH=./data/events.json

TG_BOT_TOKEN=xxxxxxxxxxxxxxxx
TELEGRAM_ALLOWED_CHAT_ID=-1001234567890

APP_TIMEZONE=Asia/Tomsk
TG_PROXY_URL=socks5://127.0.0.1:1080
APP_LOCALE=ru-RU

LOG_DIR=./logs
```

> `TELEGRAM_ALLOWED_CHAT_ID` — ID группового чата, где бот разрешён.

Данные хранятся в следующих директориях:

- события и служебные данные сохраняются в `data/events.json`;
- логи сохраняются в директории `logs/`;
- данные не теряются при перезапуске контейнера.

## REST API для фронтенда (SPA)

Backend предоставляет HTTP API для управления событиями из SPA-приложения.
API использует **ту же бизнес-логику**, что и Telegram-бот (валидация дат, TZ, формат хранения).

### Общие принципы

- формат данных: `JSON`
- хранение дат: `UTC (ISO 8601)`
- сравнение и отображение: в `APP_TIMEZONE`
- аутентификация: отсутствует (v1)
- один общий список событий

---

#### Base URL

```
/api/events
```

---

#### Структура события

```json
{
  "id": "uuid",
  "title": "Встреча",
  "description": "",
  "eventDate": "2026-01-12T11:30:00.000Z",
  "chatId": -5292908252
}
```

Поле `chatId` используется Telegram-ботом и может игнорироваться SPA.

---

### Получить все события

```
GET /api/events
```

#### Ответ

```json
{
  "events": [
    {
      "id": "uuid",
      "title": "Работа",
      "description": "",
      "eventDate": "2026-01-12T08:30:00.000Z",
      "chatId": -5292908252
    }
  ]
}
```

---

### Получить событие по id

```
GET /api/events/:id
```

#### Ответ (200)

```json
{
  "event": {
    "id": "uuid",
    "title": "Работа",
    "eventDate": "2026-01-12T08:30:00.000Z"
  }
}
```

#### Ошибка (404)

```json
{
  "error": "Not found"
}
```

---

### Создать событие

```
POST /api/events
```

#### Body

```json
{
  "title": "Работа",
  "description": "",
  "eventDate": "2026-01-12T08:30:00.000Z"
}
```

#### Ответ (201)

```json
{
  "event": {
    "id": "uuid",
    "title": "Работа",
    "eventDate": "2026-01-12T08:30:00.000Z"
  }
}
```

#### Ошибки

- `400` — неверные данные
- `400` — попытка создать событие в прошлом

```json
{
  "error": "Нельзя создать событие в прошлом"
}
```

---

### Обновить событие

```
PUT /api/events/:id
```

#### Body (частичное обновление)

```json
{
  "title": "Новая работа",
  "eventDate": "2026-01-13T08:30:00.000Z"
}
```

#### Ответ (200)

```json
{
  "event": {
    "id": "uuid",
    "title": "Новая работа",
    "eventDate": "2026-01-13T08:30:00.000Z"
  }
}
```

#### Ошибки

- `404` — событие не найдено
- `400` — неверные данные / дата в прошлом

---

### Удалить событие

```
DELETE /api/events/:id
```

#### Ответ (200)

```json
{
  "ok": true
}
```

#### Ошибка (404)

```json
{
  "error": "Not found"
}
```

---

## Валидация и бизнес-правила

- нельзя создавать и обновлять события в прошлом;
- все проверки выполняются в `APP_TIMEZONE`;
- даты в API всегда передаются и возвращаются в UTC;
- сортировка и фильтрация выполняются на стороне клиента (v1).

---

## Связь с Telegram-ботом

- события, созданные через API, **появляются в Telegram**;
- напоминания и weekly-digest работают для всех событий;
- удаление через API эквивалентно удалению через `/delete` + `/confirm`.

---

## Ограничения API v1

- нет аутентификации;
- один общий список событий;
- нет пагинации;
- нет soft-delete.
